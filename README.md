# Skills

[![skills.sh](https://skills.sh/b/Team-Discipline/skills)](https://skills.sh/Team-Discipline/skills)

AI Agent용 Skill 정의 저장소입니다. Skills는 특정 작업을 수행하는 재사용 가능한 지침(instruction) 모음으로, AI 어시스턴트가 더 효과적으로 작업할 수 있도록 돕습니다.

## Available Skills

### git-commit-split

#### 상세 설명

파일 변경사항을 **의존성(dependency)** 과 **의도(intent)** 기준으로 분석하여 Prep → Core → Fix → Polish 순서의 작고 집중된 커밋들로 분할합니다. 각 커밋은 독립적으로 revert/cherry-pick 가능해야 하며, 커밋 로그를 시간순으로 읽으면 개발자의 의사결정 흐름이 드러나야 한다는 철학을 따릅니다.

단순히 `git add -p`를 대행하는 수준이 아니라, 변경된 모든 파일을 분석하여 의존성 그래프를 그리고, 가장 안전하면서도 이야기 흐름이 있는 커밋 순서를 제안합니다. 사용자 확인 후에만 staging/commit을 진행합니다.

#### 언제 사용하면 좋은가

| 상황 | 설명 |
|------|------|
| **커밋 분할 요청** | "커밋을 분할해줘", "커밋을 나눠줘" |
| **변경사항 선별 커밋** | "이것만 커밋하고 싶어", "변경사항을 정리해줘" |
| **작업 트리 정리** | "작업 트리가 너무 지저분해", "히스토리를 정리하고 싶어" |
| **PR 리뷰 대비** | "PR 제출 전에 커밋을 깔끔하게 만들고 싶어" |
| **여러 의도가 섞인 PR** | "이 PR에 여러 기능이 섞였어, 커밋을 분리해줘" |
| **force-push 필요 상황** | "분할하려면 force-push가 필요한데 괜찮아?" |
| **Partial staging (hunk 분할)** | "한 파일 안에 버그픽스와 리팩터링이 섞여 있어" |

#### 어떻게 사용하는가

1. **자동 감지** — AI 에이전트가 관련 요청을 감지하면 자동으로 스킬이 활성화됩니다.
2. **직접 호출** — "커밋 분할해줘" 또는 "git commit split"이라고 말하면 됩니다.
3. **대화형 모드** — 기본적으로 내러티브 계획을 표시하고 그룹별로 확인을 받습니다.
4. **비대화형 모드** — "바로 진행해줘"라고 말하면 모든 그룹을 일괄 처리합니다.
5. **내러티브 모드** — "커밋 스토리를 들려줘"라고 말하면 각 커밋에 왜 이 순서인지 설명이 포함됩니다.

프로세스: Change Analysis → Build Narrative Plan → User Confirm → Validate Independence → Stage & Commit → Verify Recoverability

#### 기대 효과

- **복구 용이성 향상**: 각 커밋을 독립적으로 revert 가능 → 특정 변경만 롤백해도 나머지가 깨지지 않음
- **히스토리 가독성 개선**: "무엇을" 바꿨는지가 아니라 "왜 이 순서로" 바꿨는지가 드러남
- **코드 리뷰 효율화**: 리뷰어가 커밋 단위로 맥락을 이해할 수 있어 리뷰 시간 단축
- **cherry-pick 가능**: 특정 커밋만 골라서 핫픽스 배포나 다른 브랜치에 적용 가능
- **PR 품질 향상**: 하나의 의도 = 하나의 커밋 원칙으로 PR의 변경 범위가 명확해짐
- **회귀 최소화**: 의존성 분석을 통해 중간 상태가 깨지는 커밋을 사전에 방지
- **협업 충돌 감소**: 잘게 나누어진 커밋은 merge 시 충돌 범위가 작고 파악이 쉬움

### quality-check

#### 상세 설명

저장소 코드 품질을 **세 가지 모드**(부분 변경, 최근 커밋, 전체 감사)로 점검하고 한국어 또는 영어로 간결한 결과를 보고합니다. 코드를 절대 수정하지 않으며, 테스트도 작성하지 않고 방향만 제시합니다. 기존 린트/테스트/타입체크 도구가 있다면 우선 실행하고 그 결과를 리포트에 통합합니다.

찾은 문제는 **P1(런타임 크래시/데이터 손실/보안)**, **P2(주요 경로 회귀/API 계약 위반)**, **P3(유지보수성/중복)** 등급으로 구분하여 우선순위를 명확히 합니다.

#### 언제 사용하면 좋은가

| 상황 | 설명 | 권장 모드 |
|------|------|----------|
| **PR 제출 전 검토** | "코드 변경사항을 리뷰해줘", "PR 보내기 전에 확인해줘" | Partial |
| **작업 중간 점검** | "지금까지 한 거 품질 검사해줘" | Partial |
| **최근 작업 히스토리 점검** | "지난 5개 커밋 리뷰해줘", "최근 작업 검사해줘" | Recent Commits |
| **릴리즈 전 전체 감사** | "전체 저장소 품질 감사해줘", "릴리즈 전에 코드베이스 점검해줘" | Full Audit |
| **온보딩/코드베이스 파악** | "이 프로젝트 전체 상태를 알고 싶어" | Full Audit |
| **버그 원인 분석** | "최근 변경에서 문제가 있었던 것 같아" | Recent Commits |
| **지속적 품질 관리** | "주간 코드 품질 리포트를 만들어줘" | Full Audit |

#### 어떻게 사용하는가

1. **자동 감지** — AI 에이전트가 품질 검사 요청을 감지하면 자동 활성화됩니다.
2. **Partial 모드** — `git status --short`로 변경 파일 식별 → 해당 파일 + 직접 영향받는 계약에 한정 검사
3. **Recent Commits 모드** — `git diff HEAD~n..HEAD`로 n개 커밋 범위 검사 → 기존 도구 실행 → 커밋별/파일별 리포트
4. **Full Audit 모드** — 저장소 루트에 `QUALITY_CHECK.md` 생성 → 디렉토리별 분석 → 도구 실행 → 종합 리포트

**입력 예시:**
- "코드 품질 검사해줘" → Partial 모드 (변경사항 있으면), Full Audit (변경사항 없으면)
- "지난 10개 커밋 리뷰해줘" → Recent Commits 모드 (N=10)
- "전체 저장소 감사해줘" → Full Audit 모드

#### 기대 효과

- **회귀 조기 발견**: 커밋/PR 단계에서 잠재적 버그, 보안 취약점, API 계약 위반을 사전에 발견
- **테스트 커버리지 개선**: 구체적인 테스트 추가 지침을 제공하여 테스트 사각지대 제거
- **코드 유지보수성 향상**: 순환 의존성, 대형 함수, 불명확한 소유권 등 구조적 문제를 방향 제시
- **리뷰어 부담 감소**: 코드 리뷰 전에 quality-check를 먼저 실행하면 기본적인 문제가 정리된 상태에서 리뷰 가능
- **표준화된 품질 기준**: P1/P2/P3 등급 체계로 팀 내 일관된 품질 판단 기준 제공
- **도구 통합**: 기존 린트/테스트/타입체크를 자동 탐색하여 실행하고 결과를 리포트에 반영
- **코드 수정 제로**: 절대 코드를 수정하지 않으므로 품질 검사가 안전하게 실행됨

## Installation

```bash
npx skills add Team-Discipline/skills
```

## Usage

설치 후 AI 에이전트가 관련 작업을 감지하면 자동으로 skill을 사용합니다.

직접 호출할 수도 있습니다:
- `커밋을 분할해줘` — git-commit-split 활성화
- `코드 품질을 검사해줘` — quality-check 활성화
